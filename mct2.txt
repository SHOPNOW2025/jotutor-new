<h1 id="x_3DSTest">Test Your Integration</h1>
<p>You can validate your 3DS Authentication integration by using the 3DS Emulator provided by the payment gateway. To begin testing, log in with your test merchant profile. This profile should include the "TEST" prefix, as supplied by your payment service provider.</p>
<p>The emulator supports a variety of test card scenarios to simulate different 3DS outcomes. These test cards do not replicate the behavior of Access Control Servers (ACS).</p>
<p>The gateway supports 3DS testing for both browser-based (web authentications) and SDK/in-app channels (mobile integrations) channels. This guide covers the following topics:</p>
<ul>
  <li><a href="#x_browser3dstesting" class="x_samePageJump">Browser-based 3DS testing</a></li>
  <li><a href="#x_test_3ds_testingmobileint" class="x_samePageJump">Mobile integration 3DS testing</a></li>
  <li><a href="#x_3dsfaqs" class="x_samePageJump">FAQs</a></li>
</ul>

<h2 id="x_browser3dstesting">Test 3DS functionality: Browser integration</h2>
<p>Follow these steps to test 3DS functionality:</p>
<ol>
<li>Use a test card from the 3DS Test Cards table when submitting the <a href="../../../../../apiDocumentation/rest-json/version/latest/operation/Authentication%3a%20%20Initiate%20Authentication.html?locale=en_US">INITIATE AUTHENTICATION</a> request for your TEST merchant profile.<br/>
Use the expiry code "01/39" to get a successful authorization or payment in an end-to-end testing flow.</li>
<li>Set <code>authentication.channel = PAYER_BROWSER</code> and <code>authentication.purpose = 'PAYMENT_TRANSACTION'</code>.</li>
<li>Check if the authentication is available (<code>authentication.version</code>=3DS2).</li>
<li>Submit an <a href="../../../../../apiDocumentation/rest-json/version/latest/operation/Authentication%3a%20%20Authenticate%20Payer.html?locale=en_US">AUTHENTICATE PAYER</a> request.</li>
<li>Insert the contents of the <code>authentication.redirect.html</code> field into the page displayed to the payer, which redirects the payer's browser to the 3DS Emulator challenge page. <br/>
	The 3DS Emulator redirects the payer's browser back to your website when the authentication is complete.</li>
<li>Select a specific 3DS authentication result from the drop-down menu in the 3DS Emulator. See the values in the following Transaction Status Options in the 3DS Emulator table.
</li>	
<li>Select a successful authentication result if you want to reference this authentication transaction in a subsequent AUTHORIZE or PAY request and want the financial transaction to be successful.</li>
<li>Use the transaction ID for this 3DS authentication in the <code>authentication.transactionId</code> field of a subsequent <a href="../../../../../apiDocumentation/rest-json/version/latest/operation/Transaction%3a%20%20Authorize.html?locale=en_US">AUTHORIZE</a> or <a href="../../../../../apiDocumentation/rest-json/version/latest/operation/Transaction%3a%20%20Pay.html?locale=en_US">PAY</a> transaction request.</li>
</ol>

<h2 id="x_test_3ds_testingmobileint">Test 3DS functionality: <a href="../../../pickPaymentMethod/devicePayments/mobileintegration.html?locale=en_US">Mobile integration</a></h2>
<p>Follow these steps to test 3DS functionality:</p>
<ol>
<li>Use a test card from the following 3DS Test Cards table when submitting the <a href="../../../../../apiDocumentation/rest-json/version/latest/operation/Authentication%3a%20%20Initiate%20Authentication.html?locale=en_US">INITIATE AUTHENTICATION</a> request for your TEST merchant profile.<br/>
Use the expiry code "01/39" to get a successful authorization or payment in an end-to-end testing flow.</li>
<li>Set <code>authentication.channel = PAYER_APP</code> and <code>authentication.purpose = 'PAYMENT_TRANSACTION'</code>.</li>
<li>Check if the authentication is available (<code>authentication.version</code>=3DS2).</li>
<li>Submit an <a href="../../../../../apiDocumentation/rest-json/version/latest/operation/Authentication%3a%20%20Authenticate%20Payer.html?locale=en_US">AUTHENTICATE PAYER</a> request.</li>
<li>After the authentication is complete, the Mobile SDK receives a response:
  <ol type="a">
    <li>
    Frictionless flow:<br/> If the test card supports frictionless authentication, the Authenticate Payer response returns <code>authenticationStatus = AUTHENTICATION_SUCCESSFUL</code>, and you can proceed to step 6.
  </li>
  <li>
    Challenge flow:<br/>
If the test card requires a challenge, the Authenticate Payer response returns <code>authenticationStatus = AUTHENTICATION_PENDING</code> while the authentication process occurs. Once the challenge is completed, the Mobile SDK provides a recommendation to your application on whether to proceed with the transaction.
  </li>
</ol>
</li>
<li>Once you are ready to process your payment, use the transaction ID for this 3DS authentication in the <code>authentication.transactionId</code> field of a subsequent <a href="../../../../../apiDocumentation/rest-json/version/latest/operation/Transaction%3a%20%20Authorize.html?locale=en_US">AUTHORIZE</a> or <a href="../../../../../apiDocumentation/rest-json/version/latest/operation/Transaction%3a%20%20Pay.html?locale=en_US">PAY</a> transaction request.</li>
</ol>

<h2 id="x_3dsfaqs">FAQs</h2>
<div class="x_expand">What steps should I take when a 3DS error code is returned?</div>
<div class="x_collapse">
<p>The gateway returns error messages from the authentication servers. You can use error messages to identify which fields are causing your integration to break when an authentication request is sent. For example,</p>
<ul>
<li>Error Message:<br/>
An authentication scheme indicates that your acquirer does not onboard the authentication scheme for EMV 3DS. Contact your acquirer or PSP.
</li>
<li>Error Cause:<br/>
Before you can start using the EMV 3DS functionality, your acquirer needs to onboard you with the respective authentication scheme. Contact your acquirer or PSP to check these values. 
</li>
</ul>
</div>
<div id="x_3DSDeriveResult" class="x_expand">What should I do when there is a delay while searching for orders or transactions within the Merchant Administrator?</div>
<div class="x_collapse">
<p>You can see the authentication details in the Merchant Administrator only when the payer authentication is complete. If the payer is still going through the challenge flow or abandons the process, there can be a delay before that transaction is visible on the Merchant Administration.</p>

To see the current state of the authentication without delay, use the <a href="/api/documentation/apiDocumentation/rest-json/version/latest/operation/Transaction%3a%20%20Retrieve%20Order.html?locale=en_US">RETRIEVE ORDER</a> or <a href="/api/documentation/apiDocumentation/rest-json/version/latest/operation/Transaction%3a%20%20Retrieve%20Transaction.html?locale=en_US">RETRIEVE TRANSACTION</a> operation.
</div>
